# UI and Summary Style Guide

## Implementation Summaries
- Lead with practical effect, not technical details.
- Keep summaries to 2-4 sentences maximum.
- State what changed and why, not how.
- Non-technical language preferred.

## After UI Changes
- Always state exact UI location: card name, section, element.
- Example: "Bot Status card > Header > Status badge"
- Example: "Active Bots card > Operations list > Stop button row"

## Report Writing
- Use tables for structured data.
- Use bullet lists for findings.
- Include severity for every issue (CRITICAL / HIGH / MEDIUM / LOW).
- Include evidence (file:line, log excerpts, timestamps).
- Timestamp all updates.

## What to Avoid
- Long paragraphs of explanation.
- Repeating information the operator already knows.
- Speculative language ("might", "could possibly", "it seems").
- Trailing summaries restating what was just done.
